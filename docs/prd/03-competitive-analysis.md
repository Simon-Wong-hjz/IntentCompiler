# 2.5 竞品分析与差异化定位

## 2.5.1 竞品全景

Intent Compiler 处于"意图结构化编译"这一尚未被明确定义的品类中。当前市场上不存在直接同类产品，但有多个方向的产品在部分维度上构成替代或竞争关系。按其核心逻辑可分为以下五类：

1. **平台级 Prompt 配置能力**：ChatGPT Custom Instructions / Projects、Claude Projects
2. **Prompt 管理与增强前端**：TypingMind、Cursor Rules / Windsurf Rules
3. **Prompt 社区与模板市场**：AIPRM、FlowGPT
4. **AI 驱动的 Prompt 优化器**：PromptPerfect
5. **开发者级 Prompt 工程平台**：LangSmith Prompt Hub、PromptLayer
6. **嵌入式 AI 写作助手**：Notion AI

## 2.5.3 逐竞品差异化分析

### A. ChatGPT Custom Instructions / Projects / Claude Projects

**产品逻辑**：平台级 prompt 预配置能力，用户在自由文本框中写入全局指令，或在 Projects 中设定项目级 system prompt + 上传知识文件。

**关键差异**：
- **无结构化拆分**：本质是自由文本，不区分任务类型、约束、输出要求
- **无任务类型驱动**：不按任务类型提供差异化字段集和默认值
- **无渲染层**：写什么就是什么，不存在结构化意图到 prompt 的编译过程
- **无跨平台导出**：指令只在该平台内生效

### B. TypingMind

**产品逻辑**：增强型 AI 聊天前端，核心价值是多模型切换 + BYO API Key + Prompt Library + AI Agent 预设。

**关键差异**：
- **Prompt Library 是"存储"而非"编译"**：只保存和调用预写好的 prompt 文本
- **Agent 预设是"角色"而非"任务"**：定义角色身份，不是任务类型与约束的结构化组合
- **无 IR 层**：所有 prompt 存储为纯文本
- **无字段分层投放**

### C. AIPRM

**产品逻辑**：Chrome 插件形态的 prompt 模板社区，4500+ 社区贡献模板。

**关键差异**：
- **模板是"完整文本"而非"可编译结构"**：本质是预写好的 prompt 全文 + 少量变量占位符
- **无编译过程**
- **质量参差不齐**
- **深度绑定特定平台**

### D. PromptPerfect

**产品逻辑**：AI 驱动的 prompt 自动优化器。

**关键差异**：
- **"优化"而非"编译"**：输入和输出都是自然语言 prompt 文本
- **无结构化输入**：没有解决"用户不知道该写哪些维度"的根本问题
- **无模板体系**
- **AI 依赖度过高**

### E. FlowGPT

**产品逻辑**：社区驱动的 prompt 分享与发现平台。

**关键差异**：
- **社区市场而非构建工具**
- **无结构化层**
- **面向灵感而非面向稳定性**

### F. Cursor Rules / Windsurf Rules

**产品逻辑**：AI 编程 IDE 中的规则管理系统。

**关键差异**：
- **开发者专属**：面向程序员，不适用于通用知识工作任务
- **规则是"行为约束"而非"任务编译"**：定义的是长期约束而非本次任务的目标
- **无渲染层**
- **无任务类型区分**

**值得学习**：规则文件 + Git 版本控制的模式值得参考。

### G. Notion AI

**产品逻辑**：嵌入在 Notion 工作空间中的 AI 助手。

**关键差异**：
- **AI 是文档工具的附属功能**
- **无结构化 prompt 构建**
- **无导出为 prompt 的需求**

### H. PromptLayer / LangSmith Prompt Hub

**产品逻辑**：面向开发者和 AI 工程团队的 prompt 管理平台。

**关键差异**：
- **面向开发者而非终端用户**
- **管理"已有 prompt"而非"编译意图"**
- **无任务类型驱动的模板**

**值得学习**：版本管理、评估测试、团队协作的成熟模式可供 Phase 3 参考。

## 2.5.4 防守壁垒总结

### 壁垒 1：Prompt IR——唯一具备统一中间表示的产品

当前市场上没有竞品具备真正的"意图中间表示层"。**只有 Intent Compiler 将用户意图编译为 canonical JSON IR**，并以此为基底支撑校验、渲染、版本化和复用。竞品若要复制这一能力，需要从底层重建产品架构。

### 壁垒 2：字段分层投放——独有的渲染策略

Intent Compiler 的混合渲染模式基于字段语义将高影响字段编织进自然语言段落、将结构化约束保留在 JSON 块中。这种 **Field-Tiered Rendering** 策略在当前竞品中没有对应实现。

### 壁垒 3：任务类型驱动的模板体系——比"通用模板库"更深

Intent Compiler 的模板是"字段配置 + 默认值 + 校验规则 + 渲染策略"的结构化组合，与 IR schema 深度绑定。模板能做到：
- 根据任务类型动态显示/隐藏字段
- 注入经过设计的默认值
- 字段间冲突检测
- 同一模板生成多种风格 prompt

### 壁垒 4：无 AI 模式为主路径——更低摩擦、更可控

**零 AI 成本、零延迟**的核心流程。**确定性输出**：同样输入产出同样 prompt。**用户保持完全控制**。

### 壁垒 5：编译链路完整性——从输入到输出的全覆盖

| 链路环节 | 输入结构化 | 中间编译 | 校验/冲突检测 | 多风格渲染 | 导出/复用 |
|---|---|---|---|---|---|
| ChatGPT CI | 弱 | 无 | 无 | 无 | 无 |
| TypingMind | 弱 | 无 | 无 | 无 | 弱 |
| AIPRM | 弱 | 无 | 无 | 无 | 弱 |
| PromptPerfect | 无 | AI 黑盒 | 无 | 部分 | 弱 |
| Cursor Rules | 中 | 无 | 无 | 无 | 中 |
| PromptLayer | 中 | 无 | 无 | 部分 | 强 |
| **Intent Compiler** | **强** | **强** | **强** | **强** | **强** |

没有任何一个竞品覆盖了"结构化输入 -> IR 编译 -> 冲突校验 -> 多风格渲染 -> 导出复用"的完整链路。
